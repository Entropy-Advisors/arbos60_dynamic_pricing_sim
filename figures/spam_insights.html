2. Methodology

How a wallet gets flagged

The classification runs on every wallet that sent at least one transaction in the analysis window (2025-10-01 → 2026-03-01). For each wallet we aggregate per day first, then roll up to a single label per wallet.

  1. Per-(day, wallet) aggregation over raw_arbitrum.v_transactions (success) FULL OUTER JOIN raw_arbitrum.transactions_failed (reverts), keyed by (block_date, "from"). Reverts live in the second table — they aren't in v_transactions.success.
  2. Per-day percentile cutoffs on tx_count: p10, p01, p001. The 0.1% most-active wallets on a given day cross the p001 threshold. Cutoffs are recomputed every day so weekend vs weekday quantiles don't bleed into each other.
  3. Two independent per-day signals:
    • is_high_volume = the wallet's tx_count that day exceeds the day's p001 cutoff.
    • is_high_revert = the wallet's revert ratio that day is ≥ 30 % AND it had ≥ 50 txs that day (so a wallet with one failed tx doesn't trip the flag).
    is_spam_day = is_high_volume OR is_high_revert.
  4. Wallet-level rollup: for each address we count n_days_active, n_days_high_vol, n_days_high_rev, n_days_spam, and compute frac_spam_days = n_days_spam / n_days_active. The final tags are is_spam = (frac_spam_days ≥ 0.5) (persistent spammer) and is_spam_ever = (n_days_spam ≥ 1) (any-day flagged).

SQL: sql/arbitrum_wallet_spam_classification.sql. Fetcher: scripts/fetch_wallet_spam.py. Window-wide thresholds are exposed as placeholders (revert_min_txs, revert_ratio_threshold, spam_day_frac) so they can be tightened or relaxed without re-querying the per-day intermediate.
